# Orizon Agents
# 03. Clean Architecture

---

# Objetivo

A Orizon Agents adota a Clean Architecture para garantir um sistema desacoplado, testável e preparado para evolução contínua.

Essa arquitetura permite que regras de negócio permaneçam independentes de frameworks, banco de dados, interfaces e provedores externos.

---

# Benefícios

- Baixo acoplamento
- Alta coesão
- Facilidade para testes
- Independência do banco
- Independência do frontend
- Independência da IA utilizada
- Fácil manutenção
- Escalabilidade

---

```
src/
```

---

# Dependências

```
            Web/API
               │
               ▼
        Application
               │
               ▼
           Domain
               ▲
               │
      Infrastructure
```

Regra principal:

> As dependências sempre apontam para o centro.

O Domain nunca conhece nenhuma outra camada.

---

# Camada Domain

Responsável pelas regras de negócio.

Contém:

- Entidades
- Enums
- Value Objects
- Interfaces de domínio
- Eventos de domínio
- Constantes

Exemplo:

```
Agent

Conversation

KnowledgeDocument

AIProvider

AIModel

Tenant

User

Plan

Subscription
```

O Domain NÃO pode conhecer:

- Entity Framework
- ASP.NET
- PostgreSQL
- Redis
- OpenAI
- Controllers
- Views

---

# Camada Application

Responsável pelos casos de uso.

Contém:

- Services
- DTOs
- Interfaces
- Commands
- Queries
- Validators
- Mapeamentos

Exemplo:

```
CreateAgent

UpdateAgent

DeleteAgent

ExecuteAgent

SendMessage

GenerateEmbedding
```

---

# Camada Infrastructure

Implementa tudo que é externo.

Exemplos:

- PostgreSQL
- Redis
- OpenAI
- Gemini
- Anthropic
- SMTP
- WhatsApp
- Storage

Também contém:

- Repositories
- DbContext
- Configurations
- Migrations

---

# Camada API

Responsável por disponibilizar APIs REST.

Contém:

- Controllers
- Authentication
- Swagger
- Middlewares
- Policies

Nunca implementa regra de negócio.

---

# Camada Web

Responsável pela interface administrativa.

Contém:

- Views
- Controllers MVC
- JavaScript
- CSS
- ViewModels

Toda regra fica na Application.

---

# Camada Workers

Executa tarefas assíncronas.

Exemplos:

- Processar filas
- Enviar WhatsApp
- Enviar Emails
- Gerar Embeddings
- Sincronizar Dados
- Limpeza de Logs

# Repository Pattern

Cada agregado possui seu repositório.

Exemplo:

```
IAgentRepository

IConversationRepository

IAIProviderRepository

IKnowledgeRepository
```

---

# Dependency Injection

Toda implementação será registrada no DI.

Exemplo:

```
IAgentService

↓

AgentService

↓

IAgentRepository

↓

AgentRepository
```

---

# Unit of Work

As alterações serão persistidas em uma única transação.

```
Repository

↓

UnitOfWork

↓

SaveChanges()
```

---

# SOLID

Toda implementação seguirá:

S — Single Responsibility

O — Open Closed

L — Liskov

I — Interface Segregation

D — Dependency Inversion

---

# Clean Code

Diretrizes:

- Métodos pequenos
- Classes pequenas
- Nomes claros
- Poucos parâmetros
- Evitar duplicação
- Código legível

---

# Testabilidade

Cada camada poderá ser testada isoladamente.

Tipos de testes:

- Unitários
- Integração
- End-to-End

---

# Conclusão

A Clean Architecture garante que a Orizon Agents permaneça organizada mesmo após centenas de funcionalidades, permitindo crescimento contínuo sem perda de qualidade.
